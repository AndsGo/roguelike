## 我用了什么工具

一句话：**Claude Code + 两个 MCP 插件**。

**Claude Code** 是 Anthropic 出品的命令行 AI 编程助手。你在终端里跟它聊天，它直接帮你读文件、写代码、跑命令、提交 Git。不需要复制粘贴，不需要来回切换窗口。

但 Claude Code 真正强大的地方，是它可以接入 **MCP（Model Context Protocol）** —— 一种让 AI 连接外部工具的协议。

我只配了两个 MCP 插件：

| 插件 | 作用 | 为什么重要 |
|------|------|------------|
| **Chrome DevTools MCP** | 让 AI 能操控浏览器 | AI 可以自己打开游戏、截图、点按钮、检查 Bug |
| **Context7 MCP** | 让 AI 能查最新文档 | AI 写代码时能实时查 Phaser 引擎的最新 API |

就这两个插件，让 AI 从"只能写代码"变成了"能自己测试、自己调试、自己查文档"。

**你不需要了解这些插件的技术细节。** 你只需要知道：给 AI 配好工具，就像给新员工配好电脑和权限一样。工具越好，它能帮你做的事就越多。

## 3 天发生了什么

### Day 1 晚上：从一句话到可玩的游戏

我的第一条消息：

> "一起头脑风暴，我想开发一个类似英雄竞技场的游戏"

就这一句话。

Claude 先问了我几个问题：想要什么风格？技术上有没有偏好？大概想做多大的规模？

我回答完后，它产出了一份完整的游戏设计文档，然后问我："这个方案可以吗？"

我说可以。

接下来，它自己组建了一个 **"AI 开发团队"**：

```
项目负责人 (Lead)
  ├── 战斗工程师 → 负责战斗/元素/连击系统
  ├── 内容设计师 → 负责英雄/敌人/技能/物品数据
  ├── UI 工程师  → 负责界面/特效/动画
  └── 系统工程师 → 负责存档/成就/难度/教程
```

4 个 AI "员工"同时开工，各写各的模块，互不干扰。

**当晚结束时，游戏已经可以从头玩到尾了。**

53 个文件，7,500 行代码。从菜单进入、选英雄、走地图、打怪、逛商店、触发事件、打 Boss、胜利/失败 —— 完整闭环。

我全程只发了两条消息。

---

### Day 2：像产品经理一样提需求

第二天开始，我的角色从"甲方"变成了"产品经理"。

我不需要告诉 AI 怎么写代码。我只需要告诉它：**我试玩后觉得哪里不对。**

比如我发现按钮有时候点不到，我就说：

> "鼠标放在按钮上不变手势，这个交互需要优化"

Claude 自己去查了所有按钮的代码，发现命中区域太小，然后统一扩大了所有按钮的点击范围，还加上了鼠标手势变化。

再比如我发现游戏里中英文混杂，有些地方写的是中文，有些又是英文，我就提了一下，Claude 自己创建了一个集中管理所有文字的模块，把 19 个文件里的 150 多个英文字符串全部替换成了中文。

**最强的一招是让 AI 自己测试自己的游戏。**

我说：

> "使用 Chrome MCP 分析游戏 —— 交互、画面设计、面板显示"

Claude 通过浏览器插件打开了游戏，像真人玩家一样逐个场景检查，截图分析，最后给我一份报告：36 个 Bug，按严重程度分级。

然后我说："修复这些问题。"

它又组了一个 4 人修复团队，并行修复，全部搞定。

这就是 Day 2 的日常：**我试玩、提反馈、确认方向，AI 负责执行、测试、修复。**

这一天，它完成了：
- 258 个自动化测试
- 36 个 Bug 修复
- 全部中文化
- 5 轮性能优化
- 新增音频系统、手动技能系统、5 个新英雄、10 个新事件
- 成就面板、帮助面板、设置页面

### Day 3：打磨、部署、上线

第三天主要是打磨细节。

我玩了一局游戏，发现战斗中有些音效缺失，就说：

> "请检查战斗音效是否正常，目前测试好像战斗音效有缺失"

Claude 追踪了整个音频事件链路，发现有 3 条代码路径绕过了音效触发。它不仅修复了这个问题，还顺手补上了胜利和失败的音效。

我又发现一个逻辑 Bug：事件中的"属性加成"效果其实从来没有生效过。我说：

> "请检查事件中的属性加成的逻辑"

Claude 把整条数据链路追踪了一遍，发现属性加成只是被转换成了经验值，从来没有真正加到英雄属性上。然后它修改了 6 个文件，从底层的数据结构到顶层的 UI 显示，一次性全部修好。

最后我说：

> "将项目推送到 GitHub 并部署到 GitHub Pages"

Claude 自动配置了部署流程，推送代码，开启 GitHub Pages，几分钟后游戏就上线了。

我甚至让它通过浏览器插件自动截图，帮我写了 README 文档。

## 核心方法论：不是写代码，是管理 AI

通过这个项目，我总结出几条和 AI 协作的心法：

### 1. 你是产品经理，不是程序员

不要试图告诉 AI "怎么做"。告诉它"要做什么"和"做成什么样"就够了。

错误示范：
> "把 Button.ts 第 42 行的 pointerdown 改成 pointerup"

正确示范：
> "按钮点击经常没反应，需要优化交互体验"

后者能让 AI 自己去查代码、分析根因、全面修复。前者只能改一个点。

### 2. 给 AI 配好工具（MCP）比提升 prompt 技巧更重要

很多人纠结于怎么写出更好的 prompt。但在实际操作中，**给 AI 装好工具的效果远大于精心打磨 prompt**。

没有 Chrome MCP 的 AI，只能写代码。
有了 Chrome MCP 的 AI，能自己打开浏览器、试玩游戏、截图分析、发现 Bug。

没有 Context7 MCP 的 AI，用的可能是过时的 API。
有了 Context7 MCP 的 AI，能实时查到框架最新文档。

**工具决定了 AI 的能力上限，prompt 只是在这个上限内做微调。**

### 3. 让 AI 自己测试自己

这是我觉得最有价值的模式：

```
开发 → AI 自测 → 产出 Bug 报告 → AI 自修 → 再测 → 直到通过
```

不要等自己玩了再去反馈。直接让 AI 组一个测试团队，自己给自己找 Bug。

在这个项目里，AI 通过自测发现了 100 多个 Bug，大部分我自己玩的时候根本不会注意到。

### 4. 迭代比一步到位更有效

不要期望一次就做出完美的产品。这个游戏经历了：

- 第 1 轮：能跑起来（基础框架）
- 第 2 轮：能玩下去（修复崩溃和卡死）
- 第 3 轮：玩着舒服（交互优化、界面打磨）
- 第 4 轮：玩着有趣（新内容、新机制）
- 第 5 轮：玩着完整（音效、成就、难度、存档）

每一轮都是可运行的版本。每一轮都比上一轮好一点。

### 5. 用角色扮演激发 AI 的专业能力

当我需要专业领域的建议时，我会给 AI 指定一个角色：

> "做为一个专业的游戏音效师，对游戏音效体验提供优化建议"

AI 立刻从代码写手变成了音频顾问，给出了音高随机化、BGM 分层合成、场景音频映射等专业建议，然后自己实施了所有改进。

## 关于 MCP 配置

有朋友可能会问：MCP 配起来难吗？

说实话，**配置过程本身也可以让 AI 来帮你做**。你只需要告诉 Claude Code：

> "帮我配置 Chrome DevTools MCP 和 Context7 MCP"

它会告诉你需要安装什么、改哪个配置文件、怎么验证是否成功。

但如果你想手动了解，核心概念很简单：

MCP 就像是给 AI 装的"外挂"。裸的 AI 只能读文件、写文件、跑命令。装了 MCP 后，它能操控浏览器、查实时文档、连接数据库、调用各种 API。

目前社区已经有很多现成的 MCP 插件，覆盖了浏览器、数据库、GitHub、Figma、Slack 等常见工具。你需要的大多数场景，都已经有人做好了插件。

## 写在最后

三天前我还在纠结"要不要花几个月学游戏开发"。

三天后，一款完整的游戏已经上线，可以试玩。

**我没有学会写代码。但我学会了一件更重要的事：怎么和 AI 高效协作。**

这不是关于代码的故事。这是关于：在 AI 时代，"想做什么"比"会做什么"更重要。

你不需要会写代码，你需要的是：
1. **一个清晰的想法** —— 你想做什么
2. **一套好的工具** —— Claude Code + 合适的 MCP 插件
3. **产品经理的思维** —— 会提需求、会验收、会迭代

代码？让 AI 去写。Bug？让 AI 自己找。测试？让 AI 自己跑。部署？让 AI 自己来。

**你只需要当好那个说"做这个"和"这里不对"的人。**
