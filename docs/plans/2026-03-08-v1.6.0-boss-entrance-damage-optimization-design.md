# v1.6.0 Boss Entrance Animation + Damage Number Optimization Design

> Date: 2026-03-08
> Status: Approved
> Goal: Add cinematic boss entrance animation and optimize damage number display for clarity

## Part A: Boss Entrance Animation

### Current State
Bosses appear instantly at position. Only visual treatment: red vignette border + "BOSS" label. The 500ms `preparing` phase does nothing visual.

### Design

Use the existing 500ms prepare phase for cinematic boss entrance:

1. **Boss starts off-screen** at `x = 900` (right edge + 100px)
2. **Slide-in tween** (400ms, Cubic.easeOut): Boss slides from 900 → `ENEMY_START_X` (620)
3. **Title card overlay**: Boss name (20px, gold #ffaa00) + "BOSS" subtitle (12px, red), centered screen. Fades in 200ms, fades out 300ms after slide completes
4. **Screen shake on landing**: small shake (0.01 intensity, 200ms) when boss reaches final position
5. **Non-boss enemies** appear normally — boss slide is the focal point

### Implementation
- `BattleScene.create()`: after creating enemies, if boss battle → set boss.x = 900, tween to ENEMY_START_X
- Title card: temporary Text objects at depth 500, auto-destroyed after animation
- Landing shake via existing `BattleEffects.screenShake()`
- UnitAnimationSystem: skip idle animation for boss during slide, start idle after landing

### Files
- Modify: `src/scenes/BattleScene.ts` — boss entrance logic in create()
- Modify: `src/systems/UnitAnimationSystem.ts` — optional: delay idle start for boss
- No new files needed

## Part B: Damage Number Optimization

### Current State
Every hit creates a separate DamageNumber text. AoE combat produces 30-50 simultaneous numbers on 800×450 screen. No stacking, no overlap prevention, all unit names always visible.

### Design — 3 Improvements

### B1. Damage Number Stacking (merge rapid hits)

When multiple hits land on the same target within 150ms window, merge into one number:

- `DamageAccumulator` class (lightweight, lives in DamageSystem or new small file)
- `pendingDamage: Map<string, PendingEntry>` where PendingEntry = `{ total: number, healTotal: number, isCrit: boolean, element?: ElementType, comboCount: number, x: number, y: number, timer: number }`
- On damage event: accumulate by targetId. If first hit for this target, start 150ms timer
- On timer flush: create ONE DamageNumber with merged total. If any hit was crit, mark as crit
- Heals accumulated separately (healTotal), flushed as separate heal number
- `update(delta)` called from BattleSystem to tick timers

### B2. Damage Number Vertical Spacing (prevent overlap)

Track active damage numbers per unit to prevent stacking on same Y:

- `activeNumbers: Map<string, number[]>` — tracks Y offsets of visible numbers per unitId
- Each new number offsets upward by 18px from previous still-visible one
- Max 3 visible per unit — if 4th arrives, oldest fades early (reduce remaining duration to 200ms)
- Numbers unregister from tracking on destroy

### B3. Unit Name Visibility (reduce clutter)

- **Heroes**: Always show name labels
- **Boss enemies**: Always show name labels
- **Regular enemies**: Only show name of currently targeted enemy (being attacked or manually selected). Others show health bar only
- Implementation: in Unit class or BattleHUD, toggle nameLabel visibility based on targeting state
- Use existing TargetingSystem focus or `unit:attack` events to determine which enemy is "active"

### Files
- Create: `src/systems/DamageAccumulator.ts` — stacking + spacing logic
- Modify: `src/systems/DamageSystem.ts` — route damage numbers through accumulator
- Modify: `src/systems/SkillSystem.ts` — route skill damage numbers through accumulator
- Modify: `src/systems/StatusEffectSystem.ts` — route DoT/HoT through accumulator
- Modify: `src/entities/Unit.ts` — name label visibility toggle method
- Modify: `src/scenes/BattleScene.ts` — create accumulator, wire name visibility logic
- Modify: `src/systems/BattleSystem.ts` — call accumulator.update(delta)

## Design Principles

1. No new dependencies — use existing Phaser tween/text APIs
2. Boss entrance fits within existing 500ms prepare phase (no timing changes)
3. Damage stacking is transparent to existing systems — just intercepts number creation
4. Name visibility is a visual-only change — no gameplay impact
5. All changes testable via Vitest mocks
