> Date: 2026-03-08
> Status: Approved

## Part A: Relic RNG Determinism Fix

### Problem
`RelicSystem` uses `Math.random()` in 2 places (lines 475, 505), breaking save/load determinism. All other battle systems use injected `SeededRNG`.

### Fix
- Add `private rng: SeededRNG | null` field to RelicSystem singleton
- Modify `activateWithUnits(relics, heroes, enemies, rng)` to accept RNG parameter
- Line 475: `Math.random() > chance` → `!this.rng!.chance(chance)`
- Line 505: `Math.floor(Math.random() * length)` → `this.rng!.pick(otherEnemies)`
- BattleScene: pass `rng` to `RelicSystem.activateWithUnits()`

### Files
- `src/systems/RelicSystem.ts` — add rng field, replace 2 Math.random() calls
- `src/scenes/BattleScene.ts` — pass rng to activateWithUnits()
- `tests/systems/RelicSystem.test.ts` — verify deterministic output

## Part B: Hero Unlock Condition Diversification

### Current State
14 unlockable heroes use 3 cumulative types: `runs`, `victory`, `floor`. All reward grinding, none reward gameplay exploration.

### New Condition Types

| Type | Description | Check Point |
|------|-------------|-------------|
| `element_wins` | Win with N heroes of a specific element in party | `recordRunEnd` |
| `boss_kill` | Defeat a specific boss | `unit:kill` + MetaManager tracking |
| `no_healer_win` | Win without any healer role in party | `recordRunEnd` |
| `full_element_team` | Win with all party members sharing same element | `recordRunEnd` |
| `relic_count` | Finish a run with N+ relics | `recordRunEnd` |
| `hero_used` | Win using a specific hero in party | `recordRunEnd` |

### Data Model Changes

```typescript
// Extended unlock condition type
interface HeroUnlockCondition {
  type: 'default' | 'runs' | 'victory' | 'floor'
      | 'element_wins' | 'boss_kill' | 'no_healer_win'
      | 'full_element_team' | 'relic_count' | 'hero_used';
  threshold?: number;
  element?: ElementType;
  heroId?: string;
  bossId?: string;
  difficulty?: string;  // minimum difficulty for the condition
  description: string;
}
```

### MetaProgressionData Extension

```typescript
interface MetaProgressionData {
  // ... existing fields ...
  defeatedBosses: string[];  // NEW: boss IDs defeated across all runs
}
```

### recordRunEnd Extension

`recordRunEnd` currently receives `(victory, floor)`. Needs additional context:

```typescript
interface RunEndContext {
  victory: boolean;
  floor: number;
  partyHeroIds: string[];    // hero IDs in party
  partyElements: (ElementType | undefined)[];
  partyRoles: UnitRole[];
  relicCount: number;
  difficulty: string;
}
```

### Boss Kill Tracking

- Subscribe to `unit:kill` in MetaManager (or BattleScene)
- On boss kill: `MetaManager.recordBossKill(bossId)` → stores in `defeatedBosses`
- Check in `recordRunEnd` unlock loop

### i18n

New unlock description strings for each condition type, e.g.:
- `element_wins`: "使用2名{element}英雄通关"
- `boss_kill`: "击败{bossName}"
- `no_healer_win`: "不携带治疗者通关"
- `full_element_team`: "使用全{element}队伍通关"
- `relic_count`: "携带{n}个以上遗物完成一局"
- `hero_used`: "使用{heroName}通关"

### Backward Compatibility

- `defeatedBosses ?? []` guard in MetaManager for old saves
- Existing `runs`/`victory`/`floor` conditions unchanged for knight + shadow_assassin
- Old unlock progress preserved

## Design Principles

1. Early heroes keep simple conditions — don't block new players
2. Mid heroes introduce gameplay challenges that teach mechanics
3. Late heroes require mastery (mono-element, hard mode, boss kills)
4. Chain unlocks create discovery progression
5. Backward-compatible with existing saves
